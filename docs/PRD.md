# 雷达上位机显控平台产品需求文档（PRD）

> 文档版本：V1.0  
> 项目阶段：第一阶段——雷达显控 MVP  
> 目标平台：Windows 10/11 64 位  
> 技术栈：Qt 6.10 LTS、Qt Widgets、MSVC 2022 64 位、CMake

---

## 1. 项目概述

### 1.1 项目名称

雷达上位机显控平台。

### 1.2 项目目标

开发一套基于 Qt 6 Widgets 的 Windows 桌面端显控软件，通过 UDP 接收雷达发送的 JSON 数据，并实时展示：

- 雷达自身位置；
- 当前帧目标位置；
- 当前帧目标航迹表格；
- 当前帧目标类别统计；
- UDP 监听状态；
- 演示用模拟帧率。

平台同时支持在线高德地图和本地离线高德瓦片地图，两种地图共用同一套业务数据和选中状态。

### 1.3 第一阶段范围

第一阶段必须完成：

1. UDP 手动启动、停止和数据接收；
2. JSON 解析、字段校验和异常容错；
3. 当前帧目标数据管理；
4. 在线地图显示；
5. 离线地图显示；
6. 目标航迹表格；
7. 目标类别柱状图和饼图；
8. 地图与表格双向联动；
9. 底部分类数量和 UDP 状态展示；
10. 日志系统；
11. Python UDP 模拟数据脚本；
12. 核心逻辑自动化测试；
13. Windows 绿色发布目录和使用说明。

### 1.4 第一阶段不包含

以下功能不在第一阶段实现：

- RTSP 视频拉流；
- FFmpeg；
- YOLO 目标识别；
- ONNX Runtime；
- 历史航迹查询；
- 目标点迹或轨迹线；
- 多雷达数据融合；
- 配置持久化；
- 正式安装包；
- 自动化 UI 测试。

视频流页面仅保留占位提示，相关功能放到第二阶段。

## 2. 技术与工程约束

### 2.1 开发环境

- 操作系统：Windows 10/11 64 位；
- Qt：Qt 6.8 LTS；
- 编译器：MSVC 2022 64 位；
- 构建系统：CMake；
- 界面框架：Qt Widgets；
- 在线地图：Qt WebEngine + 高德地图 JavaScript API；
- 离线地图：QGraphicsView + 本地高德瓦片；
- 图表：Qt Charts；
- JSON：Qt JSON API；
- UDP：QUdpSocket；
- 自动化测试：Qt Test。

### 2.2 UI 开发约束

1. 不使用 `.ui` 文件；
2. 所有界面使用 C++ 代码创建；
3. 主体布局使用 `QVBoxLayout`、`QHBoxLayout`、`QSplitter`、`QStackedWidget` 等布局组件；
4. 不使用 `move()` 等绝对坐标固定业务控件位置；
5. 弹性控件使用 `QSizePolicy::Expanding`；
6. 窗口缩放时，地图、表格和图表应自适应；
7. 不依赖开发机绝对路径。

### 2.3 平台范围

第一阶段仅以 Windows 10/11 作为正式验收平台。代码应避免无必要的 Windows 专用接口，为后续 Linux 移植保留可能性，但 Linux 不属于第一阶段验收范围。

## 3. 核心业务规则

### 3.1 当前帧快照语义

每个合法 UDP JSON 报文都表示雷达当前时刻的完整目标快照。

收到一帧合法且可接受的新报文后：

- 使用本帧 `tracks` 整体替换上一帧目标集合；
- 本帧不存在的上一帧目标立即从地图、表格和统计中移除；
- 不等待目标超时；
- 不保存已消失目标历史；
- 不维护点迹；
- 不绘制轨迹线。

### 3.2 目标数量

系统不人为限制目标显示数量。报文中有多少个有效目标，就显示多少个有效目标。

典型演示场景约为 10～20 个目标，第一阶段不专门针对大规模目标场景做性能优化。

### 3.3 单雷达约束

第一阶段仅支持单雷达数据源。

- 同一 UDP 监听端口收到的合法报文均视为同一雷达发送；
- `header.frame_id` 可解析保存，但不用于多雷达分组、合并或切换；
- 不支持多个雷达同时向同一客户端提供独立目标集合。

### 3.4 坐标系

- 雷达位置和目标位置统一为 GCJ-02；
- 在线高德地图直接使用 GCJ-02；
- 离线高德瓦片直接使用 GCJ-02；
- 上位机不进行 WGS-84、BD-09 或其他坐标系转换；
- 其他坐标系应由发送端提前转换。

## 4. 总体软件架构

### 4.1 分层结构

```text
┌────────────────────────────────────────────────────────────┐
│                         MainWindow                         │
│                                                            │
│  ┌─────────────────────┐   ┌────────────────────────────┐ │
│  │      MapPanel       │   │      TrackTableView        │ │
│  │                     │   │      Config/Charts/Video   │ │
│  └─────────────────────┘   └────────────────────────────┘ │
│                                                            │
│  ┌──────────────────────────────────────────────────────┐  │
│  │                  BottomStatusBar                     │  │
│  └──────────────────────────────────────────────────────┘  │
└────────────────────────────────────────────────────────────┘
                    ▲
                    │ RadarFrame
                    │ queued signal
┌────────────────────────────────────────────────────────────┐
│                   UdpReceiver Worker Thread                │
│  QUdpSocket → JSON parse → validation → RadarFrame         │
└────────────────────────────────────────────────────────────┘
```

### 4.2 主要模块

建议至少划分以下模块：

- `MainWindow`：主界面组织和模块连接；
- `UdpReceiver`：UDP 监听、接收、序号判断和解析；
- `RadarJsonParser`：JSON 到业务模型的转换；
- `RadarFrame`：一帧完整雷达数据；
- `TrackData`：单个目标数据；
- `MapPanel`：统一地图外壳；
- `OnlineMapWidget`：在线高德地图；
- `OfflineMapWidget`：离线瓦片地图；
- `TrackTableModel`：目标表格数据模型；
- `TrackFilterProxyModel`：目标筛选和排序；
- `StatisticsWidget`：柱状图和饼图；
- `SystemConfigWidget`：UDP 和地图设置；
- `BottomStatusBar`：分类数量、UDP 状态和模拟帧率；
- `Logger`：日志输出和轮转。

### 4.3 统一业务数据流

在线地图、离线地图、表格、统计图和底部状态栏必须使用同一个 `RadarFrame` 数据源。

不允许在线地图和离线地图分别维护两套目标业务数据。

### 4.4 线程模型

UDP 接收和 JSON 解析在独立工作线程中执行。

推荐实现：

- `UdpReceiver` 继承 `QObject`；
- `UdpReceiver` 移动到专用 `QThread`；
- 工作线程随程序生命周期存在；
- 用户点击“启动监听”后，在工作线程内创建并绑定 `QUdpSocket`；
- 用户点击“停止监听”后关闭并释放 socket，但线程保持运行；
- 程序退出时安全停止线程并释放对象；
- 解析成功后通过 queued signal 向主线程发送完整 `RadarFrame`；
- 所有 QWidget、地图、表格和图表更新只能在主线程执行。

## 5. UDP 监听模块

### 5.1 功能说明

系统通过 UDP 接收雷达发送的 JSON 报文，对报文进行解析、校验和容错，再将合法帧发送到主线程展示。

### 5.2 监听配置

- 默认端口：`10000`；
- 端口控件：`QSpinBox`；
- 合法范围：`1～65535`；
- 绑定地址：`QHostAddress::AnyIPv4`；
- 不支持 IPv6；
- 不提供网卡选择；
- 程序启动后不自动监听；
- 必须由用户点击“启动监听”手动开启；
- 监听状态下禁用端口输入框；
- 停止监听后重新启用端口输入框；
- 修改后的端口在下一次启动监听时生效。

### 5.3 启停交互

系统配置页包含：

- UDP 端口输入框；
- “启动监听”按钮；
- “停止监听”按钮；
- 当前监听状态文字。

状态切换：

- 未监听：启动按钮可用，停止按钮禁用；
- 监听成功：启动按钮禁用，停止按钮可用，端口输入框禁用；
- 启动失败：恢复未监听状态并显示错误；
- 停止完成：恢复未监听状态。

### 5.4 报文示例

```json
{
  "type": "fusion_geodetic_data",
  "version": "1.0",
  "timestamp": 1773824375.675,
  "source": "imu_rtk_fusion",
  "header": {
    "frame_id": "radar_1",
    "stamp": {
      "sec": 1773824375
    },
    "sequence": 14760
  },
  "ego_position": {
    "latitude": 25.3100844,
    "longitude": 110.4111849,
    "altitude": 159.72
  },
  "target_count": 2,
  "tracks": [
    {
      "track_id": 2219,
      "type": "PEDESTRIAN",
      "position": {
        "latitude": 25.3097296,
        "longitude": 110.4157190,
        "altitude": 159.72
      },
      "distance": 5.0,
      "velocity": 6.57
    },
    {
      "track_id": 2220,
      "type": "CAR",
      "position": {
        "latitude": 25.3110000,
        "longitude": 110.4170000
      },
      "velocity": 12.34
    }
  ]
}
```

### 5.5 顶层字段处理

第一阶段不使用以下字段作为接收条件：

- `type`；
- `version`；
- `source`；
- `frame_id`。

这些字段存在时可以解析保存或写入调试日志，但不因其缺失或值不匹配而丢弃报文。

### 5.6 完整帧判定

以下情况视为整帧不可用：

- UDP 数据不是合法 JSON；
- JSON 根节点不是对象；
- 缺少 `tracks`；
- `tracks` 不是数组。

整帧不可用时：

- 丢弃该报文；
- 保留上一帧界面显示；
- 不清空地图、表格和统计；
- 写入警告日志；
- 不更新最近合法数据时间。

### 5.7 单目标容错

单个目标至少需要包含：

- `track_id`；
- `position.latitude`；
- `position.longitude`。

缺少上述任一关键字段，或字段类型无法转换时，只跳过该目标，不丢弃整帧。

非关键字段处理：

- 缺少 `type`：按“未知”处理；
- 缺少 `velocity`：界面显示 `--`；
- 缺少 `distance`：界面显示 `--`；
- 非法速度或距离：界面显示 `--`；
- `altitude`：第一阶段完全忽略，不进入业务模型，也不显示。

### 5.8 坐标校验

经纬度必须是有限数值，并满足：

- 纬度范围：`[-90, 90]`；
- 经度范围：`[-180, 180]`；
- `(0, 0)` 视为无效坐标。

处理规则：

- 目标坐标无效：跳过该目标；
- `ego_position` 无效：不更新雷达点，也不执行首次自动定位；
- 不因单个坐标异常丢弃整帧。

### 5.9 目标数量校验

若 `target_count` 与 `tracks.size()` 不一致：

- 以实际 `tracks` 数组为准；
- 记录警告日志；
- 不丢弃整帧。

### 5.10 重复航迹 ID

同一帧中若出现重复 `track_id`：

- 后出现的记录覆盖前面的记录；
- 最终只显示一个该 ID 目标；
- 记录一次警告日志。

### 5.11 目标类型映射

类型字符串先执行：

1. 去除首尾空格；
2. 转换为大写。

映射规则：

| 协议值 | 中文显示 | 地图与统计颜色 |
|---|---|---|
| `CAR` | 汽车 | 蓝色 `#3498db` |
| `TRUCK` | 卡车 | 橙色 `#e67e22` |
| `PEDESTRIAN` | 行人 | 绿色 `#2ecc71` |
| `BICYCLE` | 自行车 | 紫色 `#9b59b6` |
| `UNKNOWN` 或其他值 | 未知 | 灰色 `#95a5a6` |

第一阶段不单独增加公交车、摩托车等其他类型，未识别类型统一归入“未知”。

### 5.12 速度和距离

- 直接使用 UDP 中的原始 `velocity` 和 `distance`；
- 不根据经纬度或相邻帧重新计算；
- 数据模型保留原始 `double`；
- 缺失或非法时使用空值表示；
- 界面显示 `--`；
- 排序使用原始数值，不按格式化字符串排序。

### 5.13 序号处理

正常连续接收期间：

- 新 `sequence` 大于最近一次已接受序号：接受并刷新；
- 新 `sequence` 等于最近序号：判定为重复包并丢弃；
- 新 `sequence` 小于最近序号：判定为乱序旧包并丢弃；
- 序号发生跳跃：接受新帧并记录丢包信息。

以下情况应重置序号基准：

- 用户停止后重新启动 UDP 监听；
- 连续 3 秒没有收到合法且被接受的数据，之后重新收到数据；
- 能够判断发送端已经重启，例如新时间戳明显晚于上一帧但序号重新从小值开始。

重置后，当前合法报文应被接受。

### 5.14 UDP 状态

底部只显示一个 UDP 状态，不单独显示“雷达在线状态”。

状态规则：

- 红色：UDP 未启动或监听失败；
- 黄色：UDP 已启动，但尚未收到合法数据，或连续 3 秒没有收到合法数据；
- 绿色：UDP 已启动，且最近 3 秒内收到过合法且被接受的数据。

## 6. 业务数据模型

### 6.1 RadarFrame

一帧数据至少包含：

- 接收时间；
- 可选发送端时间戳；
- 可选序列号；
- 可选雷达位置；
- 当前帧有效目标列表；
- 当前帧分类统计结果。

### 6.2 TrackData

单个目标至少包含：

- `track_id`；
- 标准化目标类型；
- 经度；
- 纬度；
- 可选速度；
- 可选距离；
- 本轮连续出现的首次本地接收时间。

### 6.3 首次出现时间

表格“时间”字段使用上位机本地时间，不使用发送端 `timestamp`。

规则：

- 目标首次出现在连续帧序列中时记录本地时间；
- 目标持续出现在后续帧时，该时间保持不变；
- 目标从某一帧消失后，其状态立即删除；
- 相同 `track_id` 以后再次出现时，重新记录首次出现时间；
- 显示格式：`HH:mm:ss`。

## 7. 地图模块

### 7.1 功能说明

地图模块展示雷达自身位置和当前帧目标位置，支持在线地图与离线地图切换。

### 7.2 MapPanel 统一接口

`MapPanel` 作为地图统一外壳，对上层暴露一致接口，例如：

- 设置完整当前帧；
- 设置地图中心；
- 设置缩放级别；
- 选择目标；
- 清除选择；
- 切换在线或离线模式；
- 获取当前中心和缩放；
- 定位雷达；
- 目标点击信号。

内部使用 `QStackedWidget` 或 `QStackedLayout` 切换：

```text
MapPanel
├── OnlineMapWidget
└── OfflineMapWidget
```

### 7.3 地图默认状态

- 默认模式：在线地图；
- 默认图层：街道图；
- 默认中心经度：`110.416819`；
- 默认中心纬度：`25.311724`；
- 默认缩放级别：`17`；
- 允许缩放范围：`15～19`；
- 不保存用户上次地图状态。

### 7.4 首次雷达定位

程序启动后先使用默认中心。

第一次收到包含合法 `ego_position` 的有效帧后：

- 自动将地图中心移动到雷达位置；
- 显示雷达黑色点位；
- 只自动定位一次。

后续帧：

- 只更新雷达点位置；
- 不持续强制移动地图中心；
- 用户可在系统配置页点击“定位雷达”重新回到雷达位置。

### 7.5 雷达显示

- 雷达与目标使用相同的圆点视觉形式；
- 雷达颜色为黑色 `#000000`；
- 雷达点直径约 10 px；
- 雷达不进入目标表格；
- 雷达不参与柱状图、饼图或底部数量统计；
- 雷达不显示历史点迹。

### 7.6 目标显示

- 只显示当前帧目标；
- 不显示历史点迹；
- 不绘制轨迹线；
- 当前目标使用直径约 10 px 的实心圆；
- 颜色按目标类型映射；
- 选中目标增加高亮外圈，整体约 14 px；
- 标记保持固定屏幕像素大小，不随地图比例无限缩放。

### 7.7 地图标记更新

业务数据按整帧替换，但地图显示层按 `track_id` 增量更新：

- 本帧仍存在：移动已有标记并更新信息；
- 本帧新出现：创建标记；
- 本帧已消失：删除标记；
- 不先清空全部标记再重建，避免闪烁。

### 7.8 地图目标详情

地图默认不为每个目标常驻显示文字。

鼠标悬停或点击目标时显示：

- 航迹 ID；
- 类别；
- 经度；
- 纬度；
- 速度；
- 距离；
- 首次出现时间。

选中目标可在标记旁常驻显示航迹 ID。

### 7.9 表格到地图联动

点击表格目标后：

- 地图平移并以该目标为中心；
- 不改变当前缩放级别；
- 高亮目标；
- 后续目标移动只更新标记位置；
- 不持续自动跟随或强制移动地图；
- 目标消失后清除选中和高亮。

### 7.10 地图到表格联动

点击地图目标后：

- 设置该目标为当前选中目标；
- 表格选中对应 `track_id` 行；
- 若该行不在可见区域，自动滚动到该行；
- 不修改当前筛选条件；
- 若目标被筛选隐藏，只高亮地图目标，不自动切换筛选条件。

### 7.11 地图模式切换状态保持

在线与离线地图切换时必须保持：

- 当前帧目标数据；
- 雷达位置；
- 当前中心坐标；
- 当前缩放级别；
- 当前选中目标；
- 当前目标高亮状态。

切换时不返回默认中心。

## 8. 在线地图

### 8.1 技术方案

- 使用 `QWebEngineView`；
- 加载本地 HTML 页面；
- HTML 页面调用高德地图 JavaScript API；
- 使用 `QWebChannel` 完成 C++ 与 JavaScript 双向通信；
- C++ 维护业务状态；
- JavaScript 只负责地图渲染和目标点击事件回传。

## 10. 目标航迹表格

### 10.1 功能说明

表格展示当前最新一帧的全部有效目标，支持实时刷新、类型筛选、列排序和地图联动。

### 10.2 技术实现

采用：

- `TrackTableModel : public QAbstractTableModel`；
- `QSortFilterProxyModel` 或其派生类；
- `QTableView`。

不使用 `QTableWidget`。

`QTableView` 不嵌套 `QScrollArea`，直接使用自身水平和垂直滚动条。

### 10.3 表格字段

| 列 | 字段 | 显示规则 |
|---|---|---|
| 1 | 序号 | 当前筛选和排序后的可见行号，从 1 开始 |
| 2 | 航迹 ID | 显示 `track_id` |
| 3 | 类别 | 中文类别和彩色 Badge |
| 4 | 经度 | 保留 6 位小数 |
| 5 | 纬度 | 保留 6 位小数 |
| 6 | 速度（m/s） | 保留 2 位小数，缺失显示 `--` |
| 7 | 距离（m） | 保留 2 位小数，缺失显示 `--` |
| 8 | 时间 | 本轮首次本地接收时间，格式 `HH:mm:ss` |

不增加海拔列。

### 10.4 数据更新

每收到一帧合法新数据：

- 表格内容以当前帧为准；
- 新目标加入；
- 已消失目标删除；
- 仍存在目标更新最新数据；
- 不保存历史记录；
- 不分页；
- 直接滚动显示当前全部目标。

表格底部显示：

```text
当前目标：X
```

### 10.5 默认排序

- 默认按航迹 ID 升序；
- 用户可点击表头排序；
- 至少支持航迹 ID、速度、距离和时间排序；
- 新帧到来后保持当前排序字段和升降序。

### 10.6 缺失值排序

速度或距离缺失时：

- 升序：有效值从小到大，缺失值放末尾；
- 降序：有效值从大到小，缺失值仍放末尾；
- 缺失值不当作 0；
- 比较使用可选数值而不是显示字符串。

### 10.7 类型筛选

筛选框选项：

- 全部类型；
- 汽车；
- 卡车；
- 行人；
- 自行车；
- 未知。

新帧到来后保持当前筛选条件。

### 10.8 选中规则

- 表格只读；
- 整行选择；
- 单选；
- 点击任意单元格选中整行；
- 不支持多选；
- 不允许编辑；
- 单击和双击不设置两套不同业务行为。

实时刷新时：

- 保留筛选条件；
- 保留排序方式；
- 尽量保留滚动位置；
- 按 `track_id` 保留当前选中目标；
- 选中目标仍存在时继续选中；
- 选中目标消失时清除选择。

### 10.9 列宽和滚动

- 序号、类别、速度、距离、时间使用合适的固定或最小宽度；
- 航迹 ID、经度、纬度允许弹性拉伸；
- 用户可调整列宽；
- 不允许用户改变列顺序；
- 宽度不足时使用水平滚动条；
- 行数过多时使用垂直滚动条。

第二阶段可在第一阶段稳定完成后规划：

- RTSP 地址配置；
- 视频连接和断开；
- FFmpeg 拉流和解码；
- 视频帧显示；
- YOLO ONNX Runtime 推理；
- 视频检测结果与雷达目标关联；
- 视频线程、解码线程和推理线程模型；
- 视频断流重连；
- 硬件加速；
- 推理性能和延迟验收。

第二阶段必须单独编写需求和技术设计，不在第一阶段代码中提前引入无用依赖。保存为`PRD-phase2.md`。

## 22. 最终约束摘要

本阶段系统遵循以下核心原则：

> UDP 每来一帧合法完整快照，就立即以该帧 `tracks` 更新当前显示；界面只反映当前帧，不推断、不补全、不累计、不保存历史。
