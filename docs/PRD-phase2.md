# UTMS 第二阶段：视频、登录与系统监控需求文档

> 文档状态：规划中  
> 依赖：`docs/PRD.md`（第一阶段基线）、`docs/code-standards.md`、`CONTEXT.md`

---

## 1. 背景与目标

第一阶段已完成单雷达 UDP 当前帧显控。第二阶段在不改变该当前帧语义的前提下，接入单路 RTSP 视频预览、YOLO ONNX 目标检测、RTSP 原始视频录制、启动登录和本机系统监控。

本阶段不进行雷达—视频关联；视频检测结果不是雷达目标，不进入雷达表格、统计或地图。

## 2. 已确认范围

### 2.1 视频源范围

- 第二阶段首版只支持**单雷达对应单路 RTSP 视频流**。
- 不提供从界面打开本地视频文件的输入方式；操作员的视频输入仅为 RTSP。
- 不包含多路摄像头、摄像头切换、跨镜头跟踪或多雷达视频融合。

### 2.2 运行资源与依赖边界

- FFmpeg 与 ONNX Runtime 是候选第三方运行依赖，位于 `third_party/`，仅由实际使用它们的模块按需链接。
- YOLO ONNX 模型、类别文件和模型参数属于应用运行资源，位于 `models/`，不属于 `third_party/`。
- 第二阶段首版的验收模型固定为 `models/yolo26/yolo26n.onnx`，模型家族为 YOLO26；其类别文件与运行参数由同目录的 `classes.txt` 和 `model.json` 定义。
- 登录 Logo 使用仓库中的 `resources/images/guet_logo.svg`，并通过 Qt 资源路径 `:/images/guet_logo.svg` 加载；不得依赖开发机绝对路径。

### 2.3 推理执行基线

- CPU 推理必须可用，并作为首版验收基线。
- GPU 或其他硬件加速仅可作为兼容环境下的可选优化；其可用性不影响首版的功能验收。

### 2.4 RTSP 连接与恢复

- RTSP 地址输入框默认显示 `rtsp://192.168.1.101:8554/camera_1`，且允许操作员在本次运行内编辑。
- RTSP 地址只在本次运行内保留；应用退出后不持久化保存，以避免明文保存可能包含的访问凭据。
- 首版 RTSP 拉流固定使用 TCP 传输，不提供 TCP / UDP 传输方式选择。
- 应用启动后不自动连接 RTSP；操作员必须在视频流 Tab 手动发起连接。
- 操作员发起连接后，系统显示连接中状态。
- 连接失败或播放中断后，只要操作员未主动断开，系统每 3 秒自动重连。
- 界面必须区分连接中、播放中、重连中和已断开状态。
- 连接失败、播放中断或操作员主动断开后，立即清除最后一帧和检测框，并在视频区域显示明确的无视频状态。
- 若断流前检测处于开启状态，自动重连成功后自动恢复检测；模型不可用时仅报告检测错误，不影响视频预览。

### 2.5 性能验收基线

- 在验收时指定的参考 PC 上，使用 640×640 YOLO26n 模型进行 CPU 推理时，检测吞吐不得低于 10 FPS。
- 从 RTSP 画面到带检测框画面的端到端延迟不得超过 500 ms。
- GPU 或其他硬件加速可以改善性能，但不替代上述 CPU 基线。

### 2.6 视频流 Tab 与检测呈现

- 视频流 Tab 提供 RTSP 连接控制和“开启检测 / 关闭检测”控件。
- 首次手动连接 RTSP 成功后，检测默认关闭；操作员按需开启推理。自动重连成功时恢复断流前的检测开关状态。
- 关闭检测时，RTSP 视频继续显示，但系统停止推理并移除检测框叠加。
- 开启检测时，仅在视频画面中叠加检测框、类别名称和置信度。
- 视频叠加类别使用中文显示：`person` 为“行人”、`bicycle` 为“自行车”、`car` 为“汽车”、`truck` 为“卡车”；其他模型输出类别统一显示为“未知”。
- 首版不提供独立检测结果表格、检测历史、类别筛选或结果导出。
- 首版不提供置信度阈值、NMS 阈值或其他模型参数的界面编辑控件；运行参数只从 `models/yolo26/model.json` 读取。

### 2.7 实时帧处理策略

- 当推理速度低于视频解码速度时，系统丢弃等待推理的旧帧，只处理最新可用帧。
- 此策略以控制端到端延迟为优先级，不保证对视频流中的每一帧执行检测。

### 2.8 线程模型

- FFmpeg 拉流、解封装与解码在专用解码工作线程执行。
- YOLO 预处理、ONNX Runtime 推理与后处理在独立推理工作线程执行。
- GUI 线程只负责操作交互、状态呈现和视频画面绘制；工作线程不得直接访问 QWidget。
- 解码线程与推理线程通过最新帧交接，不建立无界逐帧队列。

### 2.9 模型生命周期

- 应用启动和仅视频预览时不加载模型；操作员首次开启检测时按需加载 YOLO26 模型。
- 模型加载失败时保留 RTSP 预览，并明确显示检测不可用的错误状态。
- 关闭检测仅停止推理与检测框叠加，不卸载已加载的 YOLO26 模型。
- 再次开启检测时复用已加载模型，避免重新加载造成恢复延迟。
- 仅在操作员主动断开视频或应用退出时释放模型资源；短暂断流及其自动重连期间保留模型。

### 2.10 启动登录

#### 2.10.1 启动流程

- `main.cpp` 必须先初始化 `QApplication` 和日志系统，再创建并显示登录对话框。
- 仅在登录对话框 `accept()` 后创建并显示 `MainWindow`；登录前不得构造 `MainWindow`，避免提前初始化 UDP 和视频工作线程。
- 关闭或取消登录对话框时直接退出应用。

#### 2.10.2 登录界面

- 建议使用独立的 `src/ui/LoginDialog.h` 和 `src/ui/LoginDialog.cpp`，并加入 `src/ui/CMakeLists.txt`；不使用 `.ui` 文件或绝对坐标。
- 窗口标题为“GUET-UTMS 用户登录”，初始尺寸约为 `900 × 520`，具有合理最小尺寸，并在高 DPI 下保持可用。
- 使用左右分栏布局：左侧约占 45%，右侧约占 55%。
- 左侧显示 `:/images/guet_logo.svg`，保持原始宽高比且随窗口缩放不变形；Logo 下显示“桂林电子科技大学”和“GUET-UTMS 雷达显控平台”。
- 右侧显示标题“用户登录”、账号输入框、密码输入框、登录按钮和错误提示标签。

#### 2.10.3 登录规则

- 固定演示凭据为账号 `root`、密码 `123456`。
- 账号框默认填写 `root`，密码框默认留空，并使用 `QLineEdit::Password` 回显模式。
- 点击登录按钮或按 Enter 都可提交。
- 账号或密码为空时显示“请输入账号和密码”；凭据错误时显示“账号或密码错误”。
- 登录失败后清空密码框并聚焦密码框；登录成功时调用 `accept()`。
- 不打印、不记录、不保存密码；不保存登录状态；不实现注册、找回密码、记住密码、角色权限或退出登录。
- 固定凭据验证应提取为不依赖 `QWidget` 的小型验证类或纯函数，便于 Qt Test 覆盖。

### 2.11 RTSP 原始视频录制

#### 2.11.1 范围与输出

- 录像是对当前连接 RTSP 源的原始编码视频轨道进行 MP4 封装，不是录制桌面、UTMS 窗口或视频控件截图。
- 首版不包含音频、YOLO 检测框、类别和置信度文字、UTMS 界面、本地视频回放、录像列表管理、定时录像或告警触发录像。
- 默认输出目录为 `QCoreApplication::applicationDirPath()/recordings`；目录不存在时自动创建。“打开录制目录”也应确保目录存在后，使用 `QDesktopServices::openUrl()` 打开该目录。
- 默认文件名为 `UTMS_yyyyMMdd_HHmmss.mp4`。若同名文件已存在，依次使用 `UTMS_yyyyMMdd_HHmmss_01.mp4`、`_02.mp4` 等文件名，绝不覆盖已有录像。
- 每次开始录制创建一个独立文件，首版只写入视频轨道。

#### 2.11.2 视频流 Tab 交互

- 在现有 `VideoStreamWidget` 控制栏中增加“开始录制 / 停止录制”按钮、“打开录制目录”按钮、录像状态标签和录像时长显示。
- 状态至少可呈现“未录制”“准备录制”“录制中 00:00:00”“正在停止”“保存完成”和“录制失败：具体原因”。
- RTSP 未播放、连接中、重连中或已断开时，“开始录制”不可用；仅在 `kPlaying` 后可开始录制。
- 录制中按钮文字为“停止录制”；录制期间不能重复开始；停止完成后恢复“开始录制”。
- 时长从首个等待到的关键帧实际写入文件时起算，不从点击“开始录制”时起算。保存成功后显示“保存完成（HH:mm:ss）”，直至下一次录制请求。

#### 2.11.3 录制实现与模块边界

- 录制逻辑必须位于 `media` 模块，且不得在 UI 线程写文件。推荐新增 `RtspRecorder` 和 `VideoRecordingState`；其源文件加入 `src/media/CMakeLists.txt`。
- 复用 `RtspDecoderWorker` 已读取的 `AVPacket`，不允许为了录像建立第二条 RTSP 连接，不使用 `QImage` 重新编码，也不进行视频重编码。
- 仅支持将 H.264/H.265 原始视频包 remux 为 MP4。必须正确创建输出流、复制编码参数，并使用 `av_packet_rescale_ts()` 等方式转换时间戳。
- 从开始请求后的下一个视频关键帧开始写入。若 10 秒内未等到可写入的关键帧，失败并删除空文件，状态显示“录制失败：等待关键帧超时”。
- 停止时调用 `av_write_trailer()` 并安全释放全部 FFmpeg 资源。录像初始化、写包或 trailer 失败不得中断 RTSP 解码、预览或 YOLO 推理；空文件或无效输出文件必须删除。
- `RtspDecoderWorker::decode()` 是持续运行的阻塞入口。不得仅向同一线程投递 queued slot 实现启停；应通过互斥锁或原子状态保存线程安全请求，并在解码循环处理数据包时执行实际启停。不得建立无界 `AVPacket` 队列。

#### 2.11.4 控制器、状态与停止行为

- `RtspController` 至少提供 `startRecording()` 和 `stopRecording()`，并向 UI 发出 `recordingStateChanged(state, detail, output_path)` 与 `recordingDurationChanged(seconds)`。
- `VideoStreamWidget` 增加 `startRecordingRequested()` 和 `stopRecordingRequested()`；由 `MainWindow::setupVideoController()` 按现有信号槽风格连接。
- 录制状态使用职责清晰的类型，至少包含 `kIdle`、`kStarting`、`kRecording`、`kStopping`、`kError`。
- 用户主动断开 RTSP、RTSP 播放中断、进入自动重连、主窗口关闭、应用退出或解码工作线程停止时，必须停止录像并尽可能完整保存当前文件。
- 自动重连成功后不自动恢复录像、不自动创建新文件；操作员必须重新点击开始录制。

### 2.12 系统监控 Tab

- 在现有右侧 `QTabWidget` 中新增独立“系统监控”Tab，UI 层只负责控制与呈现，指标采集位于非 UI 模块（推荐 `core`）并遵守现有分层 CMake 结构。
- Tab 使用两组概览卡片或职责等价的布局，显示本机整体 CPU 使用率、本机物理内存已用量/总量/使用率、UTMS 进程 CPU 使用率和 UTMS 进程内存占用。
- 不包含逐核心 CPU、磁盘、网络、GPU、温度或风扇监控。
- 默认显示“监控未开启”。操作员点击“开启监控”后开始采样，按钮改为“停止监控”；停止时立即停止采样，清空指标为 `--` 并显示“监控已停止”，不得显示过期样本。
- 监控开启时每秒刷新一次；采样失败时仅该指标显示明确错误，且不得影响 UDP、地图、RTSP、录像或 YOLO 功能。
- 不保存监控开启状态或任何历史指标；下一次应用启动时仍默认关闭。
- 正常每秒样本不写日志，只记录监控开启、监控停止和指标读取失败等低频事件。

### 2.13 稳定性与测试边界

- 第二阶段沿用第一阶段的两小时连续运行稳定性验收。
- 验收期间必须至少一次模拟或实际触发 RTSP 断流，并验证系统按既定策略自动恢复播放且不自动恢复录像。
- 使用 Qt Test 自动覆盖模型配置读取、YOLO 后处理、最新帧策略、RTSP 状态转换、固定凭据验证、录像状态/文件命名/关键帧超时等不依赖真实摄像头的逻辑。
- 真实 RTSP 播放、解码画面、检测框呈现、录像文件可播放性、断流保存、系统监控显示、断流恢复与两小时稳定性在实际 Windows 环境中人工验收。

## 3. 与第一阶段的边界

| 第一阶段已完成 | 第二阶段新增 |
| --- | --- |
| 单雷达 UDP JSON 当前帧、地图、表格、统计、日志 | 单路 RTSP 连接、解封装、解码、视频显示与原始视频录制 |
| 当前帧整体替换；无历史轨迹 | YOLO ONNX 推理与视频检测结果呈现 |
| 视频流 Tab 占位提示 | 视频、解码、推理、录制线程协作；断流恢复与性能验收 |
| 无应用登录、无本机资源面板 | 启动登录与按需系统监控 |
| 不引入 FFmpeg、ONNX Runtime 或相关接口 | 按需引入 FFmpeg、ONNX Runtime 和模型运行资源 |

## 4. 待决策事项

当前无待决策事项。

## 5. 暂不纳入范围

- 多路 RTSP、跨镜头关联、摄像头自动发现与切换；
- 多雷达融合；
- 视频历史检索、录像回放、录像列表、录像导出、定时录像和告警录像；
- 录音、检测框或 UTMS 界面录制、屏幕录制和视频转码；
- 雷达—视频关联、相机标定、雷达点位投影和关联结果展示；
- 注册、找回密码、记住密码、角色权限、退出登录和持久化登录状态；
- 磁盘、网络、GPU、温度、风扇或逐核心 CPU 监控；
- 第一阶段雷达数据协议和当前帧业务语义的改变。

## 6. 验收要求

### 6.1 功能验收

1. 应用启动时先显示登录对话框；未成功登录时不创建 `MainWindow`，关闭或取消登录后应用退出。
2. 登录界面满足规定的左右布局、Logo 资源和高 DPI 缩放要求；`root / 123456` 可登录，空凭据和错误凭据显示规定文字且清空密码。
3. 应用启动后不自动连接 RTSP，视频流 Tab 显示可编辑的默认 RTSP 地址和手动连接入口。
4. 操作员可手动连接、断开单路 TCP RTSP 流；连接中、播放中、重连中和已断开状态清晰可见。
5. 连接失败、断流和手动断开后不保留旧画面；非手动断开的失败每 3 秒自动重连。
6. RTSP 视频在 GUI 中以保持画面比例的方式显示；视频预览不依赖检测功能。
7. 检测默认关闭。开启检测时按需加载 `models/yolo26/yolo26n.onnx`，并在画面上叠加检测框、中文类别和置信度。
8. 仅显示行人、自行车、汽车、卡车和未知五类视频检测；其他 YOLO 类别归为未知。
9. 关闭检测后继续视频预览但停止推理、移除检测框；重新开启时复用模型。断流自动恢复后，恢复断流前已开启的检测状态。
10. RTSP 地址和检测开关均不在应用退出后持久化；模型阈值不提供操作界面编辑。
11. 仅在 RTSP 播放中允许开始录像；录像输出为独立 MP4 视频文件，不含音频、检测叠加或界面内容。
12. 录像从下一关键帧实际写入后才进入“录制中”并开始计时；10 秒无关键帧时明确失败且无空文件残留。
13. 停止录像、断流、断开、重连、关闭主窗口和应用退出均能安全封尾；重连成功后不会自动恢复录像。
14. 录像失败时 RTSP 预览和检测继续可用，UI 显示具体错误，已有有效录像不被覆盖。
15. 系统监控 Tab 默认关闭，可手动开启/停止；开启后每秒显示规定的主机与 UTMS 进程指标，停止后不保留旧值。
16. 系统监控任一指标读取失败时显示明确错误，且不影响其他功能。

### 6.2 性能与稳定性验收

1. 在指定参考 PC 上，640×640 YOLO26n 的 CPU 检测吞吐不低于 10 FPS。
2. RTSP 画面到检测框叠加的端到端延迟不超过 500 ms。
3. 推理落后时遵循最新帧优先策略，不允许无界积压造成持续增大的显示延迟。
4. 连续运行 2 小时无崩溃、无明显持续内存增长和无严重界面卡顿；其中至少一次断流可自动恢复。
5. 录像和每秒系统指标采样不得导致 UI 线程阻塞或破坏上述实时视频与稳定性指标。

### 6.3 测试与交付验收

1. Qt Test 覆盖模型配置读取、YOLO 后处理、最新帧策略、RTSP 状态转换、登录验证和录像状态机的可测试逻辑。
2. 使用真实 RTSP 环境人工验证连接、显示、检测开关、录像、断流保存、画面清除、类别叠加、系统监控、性能和两小时稳定性。

## 7. 日志与错误可见性

- 记录 RTSP 连接、断开、重连、解码失败、模型加载失败、推理初始化失败和严重运行错误，并包含足够上下文。
- 记录开始录像请求、实际开始写入、输出文件路径、用户停止录像、断流或应用退出导致的停止、保存成功、FFmpeg 输出初始化失败、数据包写入失败、trailer 写入失败以及空/无效输出删除。
- 记录系统监控开启、停止和指标读取失败。
- 不记录密码，不对每个视频帧、每个正常录像包、每次正常检测或每秒正常系统监控样本产生高频日志。
- 模型加载、推理、录像或系统监控失败不应导致 RTSP 预览停止；录像失败也不应影响检测。
